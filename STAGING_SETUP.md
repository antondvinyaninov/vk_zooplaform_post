# Настройка Staging-окружения (Разработка и Тестирование)

Этот документ описывает процесс разделения проекта на два независимых контура: **Боевой (Production)** и **Тестовый (Staging / Dev)**. 
Покажите этот файл агенту в начале следующей сессии, чтобы он сразу понял архитектурную задачу.

## 📌 Зачем это нужно?
Чтобы безопасно писать и тестировать новый функционал, не боясь сломать рабочее приложение для реальных пользователей и не повредить боевую базу данных.

---

## ✅ Шаг 1: Задачи для Владельца (Ваша часть)

Перед тем как агент начнет настройку кода, вам необходимо подготовить инфраструктуру:

1. **Создать тестовое приложение ВКонтакте:**
   - Зайдите в панель разработчика ВК и создайте новое мини-приложение (например, `ЗооПлатформа (Dev)`).
   - Сохраните его `App ID` и `Secure Key` (Защищенный ключ).
   - Добавьте это тестовое приложение в какую-нибудь тестовую группу ВК, где вы будете всё проверять.

2. **Подготовить сервер в Easypanel:**
   - Создайте новый проект (Project) в Easypanel (например, `zooplatforma-dev`).
   - Создайте внутри этого проекта новую базу данных PostgreSQL (например, `zooplatforma-dev-db`). Эта база будет изолирована от боевой.
   - Создайте App (Приложение) для бэкенда и фронтенда, точно так же, как мы делали для боевого.
   - Подключите их к GitHub, но в настройках деплоя (Source) укажите ветку `dev` вместо `main`.
   - В переменные окружения (`.env`) тестового бэкенда впишите новые данные: доступы к тестовой БД, тестовые ключи ВК и токен тестовой группы.

---

## 🤖 Шаг 2: Задачи для Агента (Моя часть на завтра)

Когда вы скажете "Погнали", агент (я) должен выполнить следующие технические шаги:

1. **Создание ветки разработки:**
   - Создать новую Git-ветку `dev` от текущего состояния `main`.
   - Убедиться, что `main` защищен от прямых коммитов без проверки (по желанию).

2. **Адаптация конфигурации (Environment variables):**
   - Убедиться, что фронтенд (Vite) умеет корректно собираться под разные URL (test API vs production API) в зависимости от переменных окружения (`VITE_API_URL`).
   - Настроить CORS на бэкенде так, чтобы он пропускал запросы от тестового домена ВК.

3. **Синхронизация CI/CD:**
   - Проверить, что пуши в ветку `dev` корректно триггерят сборку в тестовом проекте Easypanel.

---

## 🔄 Как мы будем работать дальше (Пайплайн)

1. **Разработка:** Вы ставите задачу. Агент пишет код и отправляет его в ветку `dev`.
2. **Деплой:** Код автоматически улетает на тестовый сервер (`zooplatforma-dev`) и в тестовое приложение ВК. Боевое приложение работает без изменений.
3. **QA (Тестирование):** Вы открываете тестовую группу, тестируете новую кнопку/функцию.
4. **Релиз:** Если всё работает идеально, вы даете команду агенту: "Релизим!". Агент делает Merge (слияние) ветки `dev` в `main`. Код улетает на боевой сервер. Профит! 🚀

> **Команда для старта:** "Я всё подготовил по инструкции STAGING_SETUP.md, давай создавать ветку dev и настраивать конфиги!"
